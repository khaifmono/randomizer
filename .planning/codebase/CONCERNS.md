# Codebase Concerns

**Analysis Date:** 2026-03-26

## Tech Debt

**Hardcoded Mock Authentication State:**
- Issue: Multiple route pages use hardcoded `isAdmin = true` with TODO comments, bypassing actual auth context
- Files: `apps/web/src/routes/_authenticated/users.tsx:77`, `apps/web/src/routes/_authenticated/files.tsx:150`
- Impact: All admin-only features are always enabled. No real access control validation. Cannot test member-level permissions without manual code changes.
- Fix approach: Implement React Context for auth state (e.g., `AuthContext` with `useAuth()` hook). Replace all hardcoded `isAdmin` assignments. Store current user role in context.

**Type Safety Gaps with `any` Assertions:**
- Issue: Multiple `as any` type casts bypass TypeScript checking for unsafe type transformations
- Files:
  - `apps/api/src/lib/configure-open-api.ts:62` - `scalarReferenceHandler(c as any, next)`
  - `apps/web/src/routeTree.gen.ts` - Multiple `as any` casts (generated file)
  - `apps/api/src/routes/user/user.test.ts:16-17` - `as unknown as AppEnv["Bindings"]` casts
- Impact: Hidden type errors, potential runtime crashes when API contracts change. Generated file (`routeTree.gen.ts`) is acceptable but others should be fixed.
- Fix approach:
  - For `configure-open-api.ts`: Extract scalar handler to proper typed function or use type-safe wrapper
  - For `user.test.ts`: Create proper test environment type that extends AppEnv correctly
  - Leave generated file as-is (regenerate as needed)

**Untyped Component Props in Examples:**
- Issue: `component-example.tsx` uses many inline component props without explicit interfaces
- Files: `apps/web/src/components/component-example.tsx`
- Impact: Component contracts unclear, harder to refactor components safely. Prop drilling becomes fragile.
- Fix approach: Extract component showcase logic. Create component demo registry with explicit types.

**Only 1 Test File in 7600+ Lines of Code:**
- Issue: Single test file (`apps/api/src/routes/user/user.test.ts`) covers only basic GET /users endpoint
- Files: `apps/api/src/routes/user/user.test.ts` (63 lines)
- Impact: No test coverage for routes in `apps/web/src/routes/`. API handlers lack edge case testing. Refactoring is high-risk.
- Fix approach:
  - Add integration tests for all route handlers in API
  - Add component tests for page routes in web app (settings, files, tasks, users, boards)
  - Establish minimum coverage threshold (80%)

## Security Considerations

**Admin Features Accessible to All Users:**
- Risk: Settings, file deletion, user management, audit logs visible/functional for non-admin users
- Files: `apps/web/src/routes/_authenticated/settings.tsx`, `apps/web/src/routes/_authenticated/users.tsx`, `apps/web/src/routes/_authenticated/files.tsx`
- Current mitigation: None - UI rendering is conditional but no backend enforcement
- Recommendations:
  - Implement role-based access control (RBAC) in auth middleware
  - Add permission checks on backend routes
  - Return 403 Forbidden for unauthorized endpoint access
  - Validate user role on each protected operation

**Mock Audit Logs with Fake Security Events:**
- Risk: Hardcoded audit log display shows realistic-looking security events but contains no real data
- Files: `apps/web/src/routes/_authenticated/settings.tsx:46-88`
- Current mitigation: Data is clearly mock
- Recommendations:
  - Remove or clearly label as demo when deployed
  - Implement real audit logging backend that captures: IP, timestamp, action, user, result
  - Store audit logs immutably with no ability to delete

**No HTTPS/TLS Enforcement Visible:**
- Risk: Development uses HTTP locally; production TLS configuration not specified
- Files: `README.md` shows `http://localhost:8787` as API endpoint
- Current mitigation: None visible for production
- Recommendations:
  - Add wrangler.json documentation for TLS setup
  - Document CORS policy for API
  - Add rate limiting on API routes

**Placeholder Workspace Isolation Claims:**
- Risk: UI claims "Enterprise-grade isolation" but no visible multi-tenancy implementation
- Files: `apps/web/src/routes/index.tsx:78` - feature claim
- Current mitigation: None - schema suggests single-tenant design
- Recommendations:
  - Add workspace/tenant_id to all schema tables
  - Implement row-level security (RLS) policies
  - Validate tenant isolation in all queries

## Performance Bottlenecks

**Large Page Components (600+ Lines):**
- Problem: `settings.tsx` is 624 lines with multiple tabs/features in single component
- Files: `apps/web/src/routes/_authenticated/settings.tsx`
- Cause: No component splitting. Audit logs, billing info, workspace settings all in one render tree.
- Improvement path:
  - Extract `<AuditLogsSection>`, `<BillingSection>`, `<StorageSection>` as separate components
  - Use code splitting with React.lazy() for less-visited tabs
  - Reduces initial bundle size and render time

**No Visible Data Fetching Optimization:**
- Problem: All routes use mock data. No API queries visible. Component-example.tsx has massive static imports.
- Files: `apps/web/src/routes/` and `apps/web/src/components/component-example.tsx`
- Cause: Scaffold stage - no real data layer yet
- Improvement path:
  - Integrate TanStack Query (already in package.json) for server state
  - Implement request deduplication and caching
  - Add loading/error states to all data-fetching routes

**Avatar SVG Generation on Every Render:**
- Problem: User avatars use `dicebear.com` API: `https://api.dicebear.com/7.x/initials/svg?seed=${user.name}`
- Files: `apps/web/src/routes/_authenticated/users.tsx:238`
- Cause: Dynamic URL generation, external network dependency, no local cache
- Improvement path:
  - Generate initials locally in component or use server-side generation
  - Cache generated avatars for 30 days
  - Fall back to local SVG if API unavailable

## Fragile Areas

**Settings Page with Stateful Dialogs:**
- Files: `apps/web/src/routes/_authenticated/settings.tsx:450-620`
- Why fragile: Multi-step deletion dialog has 3 states (`initial`, `confirm`, `scheduled`) managed by useState. Easy to miss state in conditional rendering.
- Safe modification:
  - Extract `<DeleteWorkspaceDialog>` with its own state management
  - Use state machine pattern (e.g., xstate) for complex flows
  - Add tests for all 3 dialog state paths
- Test coverage: None visible - dialog flow untested

**Task List with Grid Layout Breakpoints:**
- Files: `apps/web/src/routes/_authenticated/tasks.tsx:106-147`
- Why fragile: Responsive grid uses conditional visibility classes for mobile/tablet/desktop. Easy to break on viewport changes.
- Safe modification:
  - Consolidate responsive logic into separate component
  - Use CSS Grid's `clamp()` for auto-sizing instead of breakpoint-specific classes
  - Add visual regression tests for breakpoints
- Test coverage: None visible

**Authentication Context Missing:**
- Files: Entire `apps/web/src/` codebase
- Why fragile: No centralized auth state. Components access auth state inconsistently (mock boolean). Adding real auth requires touching 10+ files.
- Safe modification:
  - Create `src/contexts/auth.tsx` with `AuthProvider` and `useAuth()` hook
  - Wrap root layout with provider
  - Update all routes to read from context instead of hardcoding
- Test coverage: None - auth flow untested

**Mock Data Duplicated Across Files:**
- Files:
  - `users.tsx:45-74` - mockUsers
  - `tasks.tsx:18-27` - tasks
  - `files.tsx:46-119` - mockFiles
  - `notifications.tsx:20-55` - notifications
- Why fragile: Editing user data means updating multiple files. No single source of truth.
- Safe modification:
  - Create `src/data/mocks.ts` with all mock data
  - Export and import in each route
  - Update once, affects all pages

## Scaling Limits

**No Database Query Optimization:**
- Current capacity: Unknown (mock data only)
- Limit: When real users added, unoptimized queries will slow down:
  - Listing all users with no pagination: `users.tsx`
  - Listing all tasks with no filtering: `tasks.tsx`
  - Listing all files with naive string search: `files.tsx`
- Scaling path:
  - Add pagination to all list queries (limit 50 per page)
  - Implement server-side filtering and sorting
  - Add database indexes on frequently queried columns (user_id, workspace_id, created_at)
  - Use TanStack Query's `keepPreviousData` for smooth pagination UX

**Worker CPU Timeout Risk:**
- Current capacity: Cloudflare Workers have 30-50s max request timeout
- Limit: Any query over ~5k rows will timeout. Audit log export could exceed this.
- Scaling path:
  - Implement cursor-based pagination for exports
  - Stream responses for large result sets
  - Move heavy operations to scheduled workers (batch jobs)

**No Rate Limiting:**
- Current capacity: No visible rate limiting on API routes
- Limit: DDoS or API abuse will bring down workers instantly
- Scaling path:
  - Add Durable Objects for rate limiting (Cloudflare native solution)
  - Implement per-user quotas (e.g., 1000 requests/hour)
  - Add request size limits

## Dependencies at Risk

**TypeScript Version Lock:**
- Risk: `apps/api/package.json` pins `typescript ~5.9.3`. Patch updates only, no minor/major updates.
- Impact: New TypeScript features unavailable. Potential security issues in fixed version.
- Migration plan: Update to `^5.9.3` to allow patch updates. Test quarterly.

**Older Cloudflare Workers Types:**
- Risk: `@cloudflare/workers-types ^4.20251106.1` pins to Nov 2025 release
- Impact: New D1 features, KV API updates not available. Could miss security fixes.
- Migration plan: Update to latest `^` version. Test worker scripts before deploying.

**Base UI Dependency (New/Unproven):**
- Risk: `@base-ui/react ^1.1.0` is relatively new headless UI library. Community smaller than Radix/Headless.
- Impact: Fewer Stack Overflow answers. Fewer third-party integrations.
- Migration plan: If issues arise, migrate to Radix UI (more established). Plan 5+ days for component replacements.
